# บทที่ 5: สรุปผลและข้อเสนอแนะ

## 5.1 สรุปผลการดำเนินโครงงาน
โครงงานพัฒนาระบบผู้ช่วยปัญญาประดิษฐ์เพื่อการสืบค้นข้อมูลคู่มือปฏิบัติงานเครือข่าย (SMC-RAG System) ประสบความสำเร็จตามวัตถุประสงค์ที่ตั้งไว้ในการสร้างระบบถาม-ตอบแบบอัตโนมัติ สำหรับสนับสนุนการปฏิบัติงานของวิศวกรและผู้ดูแลระบบเครือข่ายของบริษัท โทรคมนาคมแห่งชาติ จำกัด (มหาชน) (NT) ระบบได้ผสานเทคโนโลยีรูปแบบภาษาขนาดใหญ่ (Large Language Model: Llama 3) เข้ากับสถาปัตยกรรม Retrieval-Augmented Generation (RAG) แบบลูกผสม (Hybrid Architecture) 

ระบบสามารถประมวลผลข้อความสอบถาม (Query) ในรูปแบบภาษาธรรมชาติ ดึงข้อมูลที่เกี่ยวข้องจากฐานข้อมูลคู่มือพนักงาน (SMC Knowledge Base) ผ่าน ChromaDB และสร้างคำตอบเชิงเทคนิคที่มีความแม่นยำสูง พร้อมระบุแหล่งอ้างอิงต้นฉบับ (Citation Strategy) ได้อย่างถูกต้อง การนำกลไกตรวจสอบเจตนา (Intent-Aware Routing) มาใช้ ช่วยแยกระหว่างคำถามที่ต้องการคำตอบตายตัว (Deterministic Match) และคำถามเชิงความหมาย (Semantic Search) ทำให้สามารถลดภาระการประมวลผลและลดระยะเวลาการตอบสนอง (Latency) ได้อย่างมีประสิทธิภาพ

## 5.2 การวิเคราะห์ปัญหาและแนวทางการแก้ไขปัญหา
ในระหว่างการพัฒนาระบบ ผู้จัดทำพบปัญหาและความท้าทายหลักที่กระทบต่อความเสถียรของระบบ ซึ่งได้ทำการแก้ไขและวางนโยบายกำกับดูแล (Governance Policies) ดังต่อไปนี้:

1. **ปัญหาการแต่งเติมข้อมูลของโมเดลภาษา (LLM Hallucination) ในการสรุปคำสั่งอุปกรณ์ CLI:**
   - **วิเคราะห์ปัญหา:** โมเดลภาษาทั่วไปมักพยายามสร้างประโยคอธิบายแทรกกลางระหว่างชุดคำสั่ง หรือปรับเปลี่ยนรูปแบบไวยากรณ์ของคำสั่งบรรทัดต่อบรรทัด (Command Syntax) ซึ่งก่อให้เกิดความเสี่ยงสูงหากผู้ใช้นำไปคัดลอกวางในอุปกรณ์เครือข่ายจริง (Production Network)
   - **แนวทางแก้ไข:** พัฒนากลไก **Command Protection (Phase 18)** ในส่วนของ Article Interpreter เมื่อระบบตรวจพบว่าเจตนาคือการขอคำสั่งระบบ (Intent: COMMAND) และผลลัพธ์เป็นเอกสารประเภทคำสั่ง (COMMAND_REFERENCE) ระบบจะถูกบังคับให้ใช้แม่แบบเฉพาะ (`TEMPLATE_COMMAND_REFERENCE`) เพื่อสกัดคำสั่งออกมาแบบบรรทัดต่อบรรทัด (Line-by-line Extraction) โดยไม่มีการดัดแปลงหรือสังเคราะห์ข้อความแวดล้อมเพิ่มเติม

2. **ปัญหาความครอบคลุมของค่ายอุปกรณ์ (Vendor Scope Override):**
   - **วิเคราะห์ปัญหา:** ในช่วงแรกนโยบายรักษาความปลอดภัยที่ปกป้องข้อมูลระดับองค์กร (SMC Hardening) มีความเข้มงวดเกินไป ทำให้เกิดการบล็อกเอกสารที่เกี่ยวข้องกับอุปกรณ์เครือข่ายชื่อดัง (เช่น Cisco) ทั้งที่เอกสารเหล่านั้นมีเขียนคู่มือไว้ในฐานข้อมูล SMC ภายในขององค์กร 
   - **แนวทางแก้ไข:** ทำการปรับปรุงกฎ **Vendor Scope Guard** โดยเพิ่มการตรวจสอบบริบทของ URL คู่กับชื่อค่ายอุปกรณ์ หากพบว่าแหล่งอ้างอิงมาจากเครือข่ายภายใน (Internal IP Pattern เช่น `10.192.x.x`) จะถือว่าเป็นเอกสารที่ได้รับอนุญาตให้แสดงผล (Bypass Validation) คืนความแม่นยำให้ระบบโดยไม่ละทิ้งความปลอดภัย

3. **ปัญหาความกำกวมของจุดประสงค์การถาม (Intent Classification Bug):**
   - **วิเคราะห์ปัญหา:** ผู้ใช้บางรายอาจป้อนคำถามภาษาไทยที่สะกดไม่ถูกต้อง หรือใช้คำย่อที่มีโครงสร้างกำกวม ทำให้โมเดลจัดประเภทเจตนาผิดพลาด (เช่น การสอบถามวิธีตั้งค่าถูกตีความเป็นการสอบถามพจนานุกรม) นำไปสู่การค้นหาผิดฐานข้อมูล
   - **แนวทางแก้ไข:** พัฒนากลไก **Retrieval Filter (Post-processing Filtering)** โดยเพิ่มการให้น้ำหนักคะแนน (Boost Score) จากการเปรียบเทียบคำคุ้นเคย (Title Keywords) กับเจตนาของผู้คุย เช่นคำว่า "วิธี", "การ", "แก้ปัญหา" คู่ขนานกับคะแนนความเหมือนเชิงเวกเตอร์ (Cosine Similarity) ทำให้ระบบดึงเอกสารได้แม่นยำขึ้นแม้ผู้คีย์คำสั่งมาจะใช้รูปประโยคแบบย่อ


## 5.3 การดำเนินงาน
การดำเนินงานถูกแบ่งออกเป็นกระบวนการหลักๆ ที่รับส่งข้อมูลอย่างเป็นระบบ (Pipeline Process) ดังนี้:

1. **การเตรียมความพร้อมของข้อมูล (Data Ingestion & Pre-processing):** 
   - ดึงข้อมูลจากฐานข้อมูลของระบบ SMC เดิม (Joomla HTML) 
   - ทำการสกัดเฉพาะเนื้อหาที่มีสาระสำคัญ กำจัด Noise (เช่น Navigation Bar, Footer) 
   - นำข้อความยาวมาหั่นเป็นชิ้น (Fixed-size Chunking) และแปลงสภาพเป็นโมเดลเวกเตอร์จัดเก็บใน **ChromaDB** สำหรับการค้นหาความหมาย และสร้างดรรชนี **BM25** สำหรับการค้นหาแบบตรงตัว
2. **การพัฒนากลไกวิเคราะห์การสนทนา (Query Understanding Layer):**
   - พัฒนาโครงสร้าง `chat_engine.py` โดยแบ่งเป็นเฟสการกรองระดับก่อนค้นหา (Pre-flight Validation) 
   - ติดตั้งระบบแก้คำผิด (SafeNormalizer) และจัดประเภทเจตนา 14 รูปแบบ (Intent Classifier) 
3. **การพัฒนาระบบดึงข้อมูลแบบลูกผสม (Hybrid Retrieval Engine):**
   - ออกแบบระบบให้สลับการทำงาน (Routing) ระหว่างกลไกดึงข้อมูลจากหน่วยความจำทันที (Deterministic Match) ซึ่งใช้เวลาแทบจะเป็น 0 มิลลิวินาที 
   - ประสานกับกลไกค้นหาฐานข้อมูลเวกเตอร์แบบเต็มรูปแบบ (Semantic Search) เพื่อหาบริบทสนับสนุนให้การตอบสมบูรณ์ยิ่งขึ้น
4. **การควบคุมการสังเคราะห์ข้อความ (Constrained Generation):**
   - ใช้งานโมเดล Llama 3 (ผ่าน Ollama Infrastructure) ในกรอบรูปแบบคำสั่งที่รัดกุม 
   - บังคับการแสดงผลให้แสดง **หัวข้อเรื่อง** และ **ลิงก์หน้าเว็บต้นฉบับ** เสมอ เพื่อรับรองความน่าเชื่อถือของการให้คำปรึกษา


## 5.4 ผลลัพธ์การดำเนินการ
การดำเนินการแก้ไขปัญหาและการพัฒนาระบบส่งผลให้ได้สถาปัตยกรรม SMC-RAG ที่มีวุฒิภาวะ (Maturity) อยู่ในระดับโครงสร้างที่นำไปใช้ในองค์กรธุรกิจได้จริง (Enterprise-grade Ready) โดยมีมิติผลลัพธ์ดังนี้

- **ความแม่นยำและเที่ยงตรง (Accuracy & Fidelity):** ระบบมีความสามารถในการตอบเฉพาะเนื้อหาที่เป็นสับเซตของเครือข่าย (NT-SMC) ป้องกันการนำข้อมูลซุ่มเสี่ยงหรือข้อมูลที่ไม่ผ่านการกลั่นกรองจากอินเทอร์เน็ตเข้ามาผสมได้อย่างเบ็ดเสร็จ (Zero-Hallucination on Commands)
- **ประสิทธิผลของการทำงานบนหน่วยประมวลผลระยะไกล:** ระบบประหยัดทรัพยากรส่วนกลางด้วยการใช้เทคโนโลยี Deterministic Fast-path สามารถคืนรูปคำตอบประเภท รายชื่อติดต่อ หรือหมายเลขฝ่ายเทคนิค ได้ไวกว่าการใช้ LLM วิเคราะห์ซ้ำทุกขั้นตอน 
- **ยกระดับกระบวนการทำงานแบบแผนเดิม:** ลดภาระการคีย์ค้นคว้าแบบใช้คำสำคัญ (Keyword search) แบบดั้งเดิมในเว็บไซต์ที่ต้องใช้เวลาสืบค้น พลิกหน้า และเทียบเคียงเนื้อหา เปลี่ยนมาเป็นการสนทนาที่ผู้ปฏิบัติงานจะได้ข้อมูลเชิงลึกในระยะเวลาเพียงไม่เกิน 3-4 วินาที


## 5.5 งานที่ควรทำเพิ่มในอนาคต
ถึงแม้ระบบปัญญาประดิษฐ์เพื่อการสืบค้นข้อมูลคู่มือระบบนี้จะบรรลุวัตถุประสงค์ที่ตั้งไว้ แต่เพื่อให้เกิดความสมบูรณ์ในการปรับใช้อย่างแพร่หลาย สามารถต่อยอดการพัฒนาได้ในประเด็นดังต่อไปนี้:

1. **การปรับระบบจัดการความจำของประวัติการสนทนา (Multi-turn Context Handling):**
   - ปัจจุบันระบบทำงานได้ดีเลิศเมื่อตอบคำถามเป็นแบบเดี่ยว (Single-turn Query) แต่อาจสูญเสียประเด็นความเชื่อมโยงหากผู้ใช้งานพิมพ์คำสั่งแยกต่อเนื่องกันหลายประโยค ควรบูรณาการโมดูลรวบรวมบริบท (Context Buffer Window) เพิ่มเติมในอนาคตเพื่อการสนทนาที่ต่อเนื่อง
2. **การบูรณาการกับระบบสถานะอุปกรณ์แบบเรียลไทม์ (Live API Integration):**
   - เปลี่ยนบทบาทของแชทบอทจากเพียงแค่กลไกอ่านเอกสารคู่มือ (Static Document Processing) มุ่งสู่การวินิจฉัยปัญหาเชิงรุก เช่น ให้ระบบสามารถขอสิทธิ์ทดลองส่งแพ็กเกจ (Ping) วินิจฉัยเร้าเตอร์ปลายทาง แล้วนำผลลัพธ์จาก API จริงมาวิเคราะห์เพื่อสรุปหาสาเหตุอาการเสียให้ครบจบในระบบเดียว
3. **การออกแบบกระบวนงานกวาดข้อมูลอัตโนมัติประจำวัน (Nightly Automated Re-crawler):**
   - ในทุกครั้งที่ฐานข้อมูล Joomla SMC ของหน่วยงานมีการปรับปรุงเนื้อหา หรือเพิ่มบทความใหม่ การดึงข้อมูลอัปเดตระบบยังถือเป็นกระบวนการแบบ Manual ควรสื่อนำเครื่องมือ (Workflow Automation Tool) มาผูกเข้ากับ Cron Job เพื่อจัดทำดัชนีเวกเตอร์ใหม่โดยอัตโนมัติในทุกวัฏจักรเวลากลางคืน 
4. **การเสริมศักยภาพให้รองรับไฟล์รูปแบบใหม่ (Multimodal Support):**
   - พัฒนาต่อยอดพารามิเตอร์การแยกวิเคราะห์ (Parser) ให้ครอบคลุมรูปแบบ Topology Diagram (.pdf) การอ่านค่าตารางจากสเปรดชีต (Excel Sheet) ตลอดจนแผนผังการกำหนดเส้นทางจราจรเครือข่ายที่ซับซ้อน เพื่อให้อ่านและตอบบริบทภาพรวมโครงข่ายได้ลึกซึ้งยิ่งขึ้น
