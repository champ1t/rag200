## 📌 ส่วนที่ 1: นำไปแทรกใน บทที่ 2 (Literature Review)
*คำแนะนำ: นำไปต่อท้ายหัวข้อย่อยที่พูดถึงทฤษฎี Chatbot หรือ RAG ทั่วไป เพื่อแสดงให้กรรมการเห็นว่าเรามีการทบทวนวรรณกรรมและเปรียบเทียบข้อดีข้อเสียอย่างเป็นระบบ*

### 2.X การวิเคราะห์เปรียบเทียบเทคโนโลยีระบบสนทนาอัตโนมัติระดับองค์กร (Comparative Analysis of Enterprise Conversational AI)

ในการพัฒนาระบบผู้ช่วยเพื่อตอบคำถามทางเทคนิคระดับองค์กร (Enterprise Technical Support) สิ่งสำคัญที่สุดคือความน่าเชื่อถือของข้อมูล (Data Reliability) อย่างไรก็ตาม เทคโนโลยีทางเลือกต่างๆ ล้วนมีข้อได้เปรียบและข้อจำกัดที่แตกต่างกัน จากการศึกษาทบทวนวรรณกรรม สามารถสรุปเปรียบเทียบข้อได้เปรียบเชิงโครงสร้างของปัญญาประดิษฐ์แต่ละประเภท ดังตารางที่ 2.X

**ตารางที่ 2.X:** การเปรียบเทียบสถาปัตยกรรมระบบสนทนาอัตโนมัติ
| แนวทางการพัฒนา (Approach) | จุดเด่น (Strength) | ข้อจำกัด (Weakness) |
| :--- | :--- | :--- |
| **Rule-based Chatbot** | มีความแม่นยำสูง ควบคุมทิศทางคำตอบได้ 100% | ไม่ยืดหยุ่นต่อคำถามนอกสคริปต์ ไม่สามารถอธิบายความซับซ้อนได้ |
| **Pure LLM (e.g. ChatGPT)** | เข้าใจภาษาธรรมชาติได้ดีเยี่ยม ยืดหยุ่นสูง | ความเสี่ยงเกิดอาการหลอน (Hallucination) ไม่มีข้อมูลเฉพาะภายในองค์กร |
| **Vanilla RAG** | คำตอบอ้างอิงจากฐานข้อมูลจริง (Knowledge Grounded) | มีโอกาสดึงข้อมูลผิดพลาด (Retrieval Noise) หากคำถามกว้างเกินไป |
| **Hybrid RAG (Proposed)** | สมดุลระหว่างความไว (Exact Match) และความลึก (Semantic) | มีความซับซ้อนในการพัฒนาระบบคัดกรองเจตนา (Intent Filter) |

**บทวิเคราะห์จากตาราง:** 
จากข้อจำกัดของ **Vanilla RAG** แบบดั้งเดิมที่พึ่งพาเพียงฐานข้อมูลเวกเตอร์แต่เพียงอย่างเดียว ซึ่งอาจทำให้เกิดความผิดพลาดในการดึงข้อมูลที่ไม่ได้สัดส่วน โครงงานนี้จึงนำเสนอสถาปัตยกรรม **Hybrid RAG** ที่ผสานกลไกกฎข้อบังคับ (Rule Governance) เข้าครอบคลุมกระบวนการ RAG เพื่อดึงข้อดีของความแม่นยำสูง (Rule-based) มาบรรเทาจุดอ่อนของการเกิด Retrieval Noise ถือเป็นการแก้ปัญหางานบริการจัดการความรู้ (Knowledge Management) ระดับองค์กรได้อย่างสมบูรณ์

## 📌 ส่วนที่ 2: นำไปแทรกใน บทที่ 3 (Methodology / System Design)
*คำแนะนำ: นำไปแทรกเป็นหัวข้อใหญ่ใหม่ หรือเพิ่มเป็น Subsection ก่อนจบสถาปัตยกรรมระบบ เพื่อชี้แจง Design Decision ให้กรรมการเห็นความเป็นวิศวกรซอฟต์แวร์*

### 3.X เหตุผลในการตัดสินใจเลือกใช้สถาปัตยกรรมและเทคโนโลยี (Design Rationale)

ในการออกแบบระบบ SMC-RAG โครงสร้างของระบบไม่ได้ถูกเลือกใช้เพียงเพราะความทันสมัยทางเทคโนโลยีเท่านั้น แต่ได้ผ่านการประเมินเชิงวิศวกรรม (Engineering Decision) เพื่อให้สอดคล้องกับทรัพยากรที่มีจำกัดและนโยบายด้านความปลอดภัยของบริษัท โทรคมนาคมแห่งชาติ จำกัด (มหาชน) (NT) เป็นหลักเชิงวิศวกรรมการออกแบบดังนี้:

#### 3.X.1 การทำงานระบบปิด (On-Premise Deployment) และการเลือกใช้ Llama 3.2 (3B)
*   **นโยบายความมั่นคงปลอดภัยของข้อมูล (Data Privacy):** เนื่องจากคู่มือ SMC และพารามิเตอร์การตั้งค่าเครือข่ายถือเป็นความลับโครงข่ายระดับองค์กร (Network Topology Secrets) จึงไม่สามารถทรานสเฟอร์ข้อมูลผ่าน API ของคลาวด์สาธารณะ (เช่น OpenAI) ได้ ระบบจึงต้องถูกติดตั้งและขับเคลื่อนในสภาพแวดล้อมระบบปิด (On-premise Environment) ผ่านโปรแกรม Ollama
*   **เหตุผลที่เลือกโมเดล Llama 3.2 3B:** แม้จะมีโมเดลโอเพนซอร์สขนาดใหญ่ (เช่น 8B หรือ 70B) แต่จากสภาวะฮาร์ดแวร์เซิร์ฟเวอร์แบบ Local ที่ไม่ได้ประมวลผลผ่านชิปเซ็ตเชิงกราฟิกชั้นสูง การเลือกใช้สถาปัตยกรรมของ Llama 3.2 ที่มีพารามิเตอร์ขนาด 3 พันล้านพารามิเตอร์ (3 Billion) ถือเป็น **จุดสมดุลที่ดีที่สุด (Sweet Spot)** ระหว่างความเร็วในการทำ Inference เพื่อการตอบสนองแบบเรียลไทม์ และยังคงขีดความสามารถที่เพียงพอในการประมวลผลความเข้าใจทางภาษา (Natural Language Understanding) ร่วมกับบริบทเฉพาะทาง

#### 3.X.2 การเลือกใช้ฐานข้อมูลเวกเตอร์ ChromaDB (Local Vector Store Selection)
ในการเปรียบเทียบฐานข้อมูลเวกเตอร์ ปัจจัยแห่งความสำเร็จของระบบ On-premise คือความเข้ากันได้กับการเก็บสถิติท้องถิ่น (Local Persistence)
*   **ChromaDB** ถูกคัดเลือกมาใช้แทน Vector Database บนคลาวด์ตัวอื่น (เช่น Pinecone) เนื่องจากสามารถบันทึกคลังเวกเตอร์ (Embedding Vectors) ฝังลงในไดรฟ์ของเครือข่ายเอนทิตีเอง (In-memory and Local Disk) ช่วยให้แฝงอยู่ในระบบเครื่องข่ายปิดได้โดยสมบูรณ์
*   ความสามารถการดึงค่าน้ำหนักสัมพันธ์ (Cosine Similarity) ทำงานประสานกับคลังข้อมูล Sentence-transformer ได้อย่างราบรื่น ทำให้ระบบสามารถสืบค้นข้อความจำนวนมากได้อย่างรวดเร็ว

#### 3.X.3 สถาปัตยกรรมคัดกรองด่านหน้า (Deterministic-First Routing)
ปัญหาที่พบได้ทั่วไปในการสร้างระบบ RAG แบบประเมินค่าตามความหมาย (Vanilla RAG) คือการสิ้นเปลืองทรัพยากรชิปประมวลผลไปกับการสังเคราะห์ข้อความซ้ำซาก สำหรับการตอบคำถามสั้นๆ หรือเอกสารคู่มือทางเทคนิคที่ต้องการรูปแบบการตอบสนองที่ตายตัว
*   **เหตุผลในการออกแบบ:** ระบบจึงถูกตั้งค่าใช้เส้นทางนโยบาย **Deterministic-First Routing** ทำหน้าที่เป็นปราการด่านหน้า โดยหากเป็นคำถามที่ผู้ใช้งานร้องขอข้อมูลติดต่อ (Contact Lookup) หรือการสอบถามชุดคำสั่งพื้นฐาน (Command List) ที่ไม่มีความกำกวม ระบบจะส่งข้อมูลกลับไปแสดงเป็นลิงก์และคัดลอกจากฐานข้อมูลแบบตรงตัวข้ามหน้า RAG Generation ทันที (Bypass LLM) 
*   **ผลลัพธ์ที่เป็นประโยชน์:** การตัดสินใจทางสถาปัตยกรรมนี้ทำให้สามารถจัดการกับคำถามสัดส่วนกว่าร้อยละ 40 ของคำถามซ้ำซาก (Repetitive Questions) ได้ด้วยความเร็ว 0 มิลลิวินาที เป็นการสงวนทรัพยากรตัวประมวลผล (Compute Power) สำหรับคำถามเชิงเทคนิคในระดับแก้ไขสถานการณ์ (Troubleshooting) ที่ซับซ้อนกว่า ถือเป็นการบริหารจัดการทรัพยากรฮาร์ดแวร์เครือข่ายได้อย่างมีประสิทธิภาพคุ้มค่า (Resource Optimization)
